# rapid-pdf audit — 2026-06-26

Adversarial re-audit of the current code (HEAD after the `feature/perf-rendering` merge), run by three independent reviewers focused on (1) movable images / interop, (2) the recently-changed perf/rotation/undo code, and (3) the save / data-integrity path. Findings were verified against the actual source and, where possible, reproduced empirically.

## Fixed in this branch (`feature/audit-fixes`) — verified

| Fix | Was | Now | Severity |
|---|---|---|---|
| **In-place save atomicity** | `shutil.move(tmp, target)` — on Windows, overwriting an existing file falls back to a non-atomic copy+delete; a crash mid-copy corrupts the original with no `.bak`. | `os.replace(tmp, target)` — atomic overwrite on Windows + POSIX. | **High (data loss)** |
| **Closed-doc on save failure** | After `self.doc.close()`, if the move or reopen threw, `self.doc` was left pointing at a *closed* Document. The UI still thought a doc was open → every later render/save raised "document closed", unrecoverable. | `self.doc` is set to `None` immediately after close; any failure leaves it `None` so the UI treats the doc as closed and can recover. | **Critical** |
| **Undo desync on Page-menu delete** | The "undo safety" fix cleared the undo stack only in the *Organizer* delete path (`_on_pages_deleted`). The **Page → Delete Current Page** path did not, so undo could replay against removed/renumbered items. | `delete_current_page` now clears the undo stack too. | **High** |
| **Undo desync on page reorder** | Reorder re-bases every item's `page_num`; the undo stack's commands still referenced the old numbering, landing undone items on the wrong page. The reorder path never cleared the stack. | `_on_pages_reordered_perm` now clears the undo stack. | **High** |

## Confirmed but NOT fixed here — needs a design decision

### Image accumulation / double-render (CRITICAL) — all three reviewers flagged it

**Proven empirically:** baking an image annotation N times stacks N content-stream images on the page (measured 0 → 1 → 2 → 3).

**Root cause.** Images are baked into the page **content stream** via `page.insert_image()` (`core/pdf_document.py`, `write_annotations`, image branch) and are **untagged**. But the strip step (`delete_tagged_annotations`) only removes *annotations* whose `info["title"] == "rapid-pdf"`. Images are also **excluded from the embedded JSON model** (`canvas.py` `_item_to_json` returns `None` for `type == "image"`). So the strip/rebake symmetry that keeps every other markup type clean is broken for images:
- **Same session, repeated saves:** the live `ImageAnnotationItem` is never removed, so each save re-bakes it on top of the prior baked copy → accumulation, and a visible double-render (baked background copy + live overlay) until the page reloads.
- **Fresh reopen** is fine (image shows once, from the content stream, and is re-liftable).

**Why it's not auto-fixed:** the fix touches the "faithful content-stream image vs. annotation" tradeoff in the PRD and needs visual verification across viewers — not a safe blind change to a daily-use tool.

**Two options:**

1. **Bake images as tagged stamp annotations** (title == `rapid-pdf`) and include them in the JSON model. Pro: fully symmetric with all other markup — strip/rebake just works, no accumulation. Con: stamp annotations are not "native" page content; some viewers render them differently, which weakens the "faithful save / looks identical in Acrobat" promise.

2. **(Recommended) Flatten-on-save + reload.** Keep images as native content-stream XObjects (faithful), but after a successful save, drop the live image overlay and reload the page so the image comes back as an *embedded, liftable* image — exactly how a third-party PDF's images already behave. Pro: preserves faithfulness and reuses the existing lift mechanism; no accumulation (no live overlay left to re-bake). Con: an image isn't draggable immediately after save until re-lifted (consistent with reopen behavior, but a small UX change to confirm).

Recommendation: **option 2** — it matches the Acrobat-style "objects reconstructed from the content stream" model you already built and liked, and keeps saves faithful. Worth its own focused branch with hands-on GUI testing.

## Other findings (backlog — not blocking)

- **Partial bake reported as "Saved" (medium).** `write_annotations` catches per-annotation errors and only `print()`s them; `save_pdf` still reports success. A dropped image bake (e.g. an exotic clipboard image `insert_image` rejects) means the image vanishes with no warning. Fix: collect bake failures and surface a non-fatal warning.
- **Embedded-image cache vs. debounced render (medium).** `_embedded_image_at` / `_ensure_embedded_images` key off `_current_page`, which `set_page` updates *before* the 80 ms debounced render. A hit-test in that window can scan the new page while the background pixmap is still the old page. Mostly closed by the press-time `_flush_pending_render()`. Fix: key the cache on the page the `_bg_item` actually shows.
- **Continuous-scroll over-turn (suspected, medium).** A second wheel tick during the page-turn debounce can re-evaluate the edge condition against the stale scrollbar and skip a page. Fix: guard `_goto_adjacent_page` against re-entry while a turn is in flight.
- **Lifted-image rotation/shear (suspected, medium).** Lifting captures only the bounding rect, not the per-image CTM, so a rotated/sheared *image placement* (on an unrotated page) re-bakes axis-aligned. The recent rotation fix addressed page-level rotation only. Fix: capture the transform from `get_image_info(xrefs=True)`.
